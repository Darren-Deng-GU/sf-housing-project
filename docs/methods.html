Phase 1: Data Ingestion

All four datasets are downloaded programmatically using Python's requests library and the respective APIs. The raw files are then uploaded to an AWS S3 bucket using boto3, establishing S3 as our centralized data lake.

Why S3?

S3 provides durable, scalable object storage that integrates natively with both Snowflake (via external stages) and Databricks (via S3 mounts). For production use, this pattern supports automatic data refresh: a scheduled Lambda function could re-pull data from APIs daily and land new files in S3, triggering downstream ETL.

# Upload to S3 (from 01_data_ingestion.py) import boto3 s3 = boto3.client("s3", region_name="us-east-1") s3.upload_file( "data/raw/Building_Permits.csv", "sf-housing-project", "raw/Building_Permits.csv" )

Phase 2: Snowflake ETL

Snowflake serves as our primary transformation engine. We chose Snowflake because it provides a fully managed, columnar SQL warehouse that requires zero cluster management—ideal for teams that need powerful SQL analytics without DevOps overhead.

Infrastructure Setup

We created a dedicated database (SF_HOUSING) with two schemas: RAW for staging and ANALYTICS for transformed data. An external stage connects directly to our S3 bucket, enabling COPY INTO commands that load CSVs into Snowflake tables in seconds.

-- Load data from S3 into Snowflake COPY INTO RAW.BUILDING_PERMITS FROM @RAW.S3_STAGE/Building_Permits.csv FILE_FORMAT = RAW.CSV_FORMAT ON_ERROR = 'CONTINUE';

Transformations

The ANALYTICS schema contains cleaned tables and analytical views. Key transformations include:

Date Parsing

Convert string dates to timestamps using TRY_TO_TIMESTAMP for safe parsing with error handling.

Processing Time

Calculate days between filing and issuance using DATEDIFF, filtering outliers beyond 1,000 days.

Aggregation Views

Pre-computed views for permits trend, neighborhood rankings, change-of-use flows, and cross-analysis joins.

Cross-Dataset Join

Join permit data with Census income/rent data by geography to explore the relationship between neighborhood wealth and permit processing times.

Why Snowflake over Pandas?

While this dataset fits in memory, Snowflake demonstrates a production-ready pattern. If we scaled from one city to all 50 states (~50M+ permits), Snowflake would handle the volume without code changes—just larger warehouse sizes. The SQL-based approach also makes the pipeline accessible to analysts who don't write Python.

Phase 3: PySpark Analysis (Databricks)

As a complementary analysis, we processed the permits dataset using PySpark on Databricks to demonstrate distributed computing capabilities. This step is particularly relevant for a "what if" scenario: if this analysis were expanded to cover all U.S. cities, Spark's distributed execution would be essential.

# Window function: rank neighborhoods by year neighborhood_ranked = spark.sql(""" SELECT neighborhood, issued_year, permit_count, RANK() OVER ( PARTITION BY issued_year ORDER BY permit_count DESC ) AS rank_in_year FROM ( SELECT neighborhood, issued_year, COUNT(*) AS permit_count FROM permits GROUP BY neighborhood, issued_year ) """)

Phase 4: Visualization & Delivery

Processed data is exported from Snowflake as CSV files and fed into a Python visualization pipeline using Plotly (interactive charts) and Folium (geographic maps). Each chart is exported as a standalone HTML file and embedded in this GitHub Pages website.

Why Plotly + GitHub Pages?

This combination gives us rich interactivity (hover, zoom, pan) without requiring a server. Every visualization loads directly in the browser, making the analysis accessible to non-technical stakeholders—a key requirement for policy communication.
